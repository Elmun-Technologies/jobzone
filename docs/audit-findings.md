## 1. Executive Summary

### Overall health
The app is well-structured (Riverpod + go_router + a token-based design system + Supabase RLS), and the team clearly understands the patterns ‚Äî there are exemplary implementations of almost everything (`JzErrorState` with retry, `is_job_owner` RLS gating, `applicationStatusColor` tokens, the avatar upload flow). **The problem is consistency, not capability.** The same correct pattern exists somewhere in the codebase while a dozen sibling screens reimplement a broken version of it.

### The biggest recurring themes (by frequency and blast radius)

1. **PII / location over-exposure via permissive RLS (the dominant P0 theme).** Five separate P0 findings converge on one root cause: `profiles selectable by authenticated using (true)` plus a `security_invoker` `profiles_public` view that projects raw `lat/lng`, `desired_pay`, phone, email, bio, and home address. Any logged-in account can harvest every worker's exact home coordinates and contact info. `contact_info` and `worker_reviews` have the same `using (true)` leak. **This is the single most important thing to fix.**

2. **Privilege escalation through under-segmented write policies.** Applicants can self-promote to `hired`/`offer` via both the `application_status_history` INSERT policy *and* a column-unguarded `applications` UPDATE policy. Anyone can self-join an arbitrary conversation, post unsolicited company reviews, or pollute the shared `skills` catalog. The hardening migration (0010) fixed *some* of these and missed the rest ‚Äî a tell-tale "audit pass that didn't finish."

3. **Fail-open / fail-dangerous edge functions.** Every `if (secret && header !== secret)` gate silently disables auth when the env var is unset; `meili-reindex`, `search-jobs`, and `telegram-webhook` have weak or no auth; raw internal error strings leak to callers.

4. **Dead controls and discarded user input (UX correctness).** A recurring, trust-eroding pattern: the Apply form collects name/email/CV and **silently discards all three**; "Withdraw Application," "Delete Account," composer attach/mic, gallery "See all," company website link, and the home location selector all look interactive but do nothing. The CV upload box is the worst ‚Äî a core marketplace artifact dropped on the floor.

5. **Missing error/loading states & silent failures.** ~22 screens render `Center(Text(errUnknown))` with no retry; many write actions (`updateStatus`, chat send, add-person, settings save) have no try/catch, so failures are invisible or hang a button forever. Raw `PostgrestException.toString()` is shown to users in ~28 places.

6. **Routing gates broken for whole user classes.** Employers can't open chat/calls (guard bounces them off `/chat`); password reset is unreachable for onboarded users; apply‚Üísuccess destroys the back stack.

7. **Design-system & i18n drift.** Brand gold `#FFC629`/ink duplicated across 10+ files (no token, breaks dark mode); hardcoded English in salary suffixes, relative times, "Present," city lists, gender/position dropdowns; 9 duplicate ARB keys silently overriding casing intent.

## All findings


### P0

- **createCompany is not idempotent and owner_id has no unique constraint ‚Äî myCompany().maybeSingle() can crash** `[data]` - `lib/features/employer/data/company_admin_repository.dart:20-31 and 34-73`
  - myCompany() queries companies by owner_id and calls .maybeSingle(), which throws a PostgrestException if more than one row matches. createCompany() never checks for an existing company and always inserts a new row, and companies.owner_id has NO unique constraint (migrations/0002_jobs_companies.sql:24 is a plain nullable FK). Any path that reaches create-company twice (e.g. setProfileComplete failing after a successful insert in create_company_page.dart:59, then a retry) produces a second company, after which every myCompany() read ‚Äî i.e. dashboard, company tab, people, gallery ‚Äî throws and renders the generic error state, bricking the employer area.
  - _Fix:_ Make createCompany idempotent: re-fetch myCompany() first and return it if present, or use upsert on owner_id. Add a unique index on companies(owner_id) in a migration. Defensively, use .limit(1).maybeSingle() or order+first in myCompany().
- **Secret gates are fail-OPEN: every `if (secret && header !== secret)` skips auth entirely when the env var is unset** `[security]` - `supabase/functions/send-notification/index.ts:15-18`
  - The shared pattern across send-notification, push-dispatch, notify-dispatch, meili-sync, and generate-job-content is `const secret = Deno.env.get('EDGE_SHARED_SECRET'); if (secret && req.headers.get('x-edge-secret') !== secret) return 401`. If EDGE_SHARED_SECRET is not configured, the guard is skipped and the endpoint is unauthenticated. send-notification is the worst case: it is verify_jwt=false (config.toml:59-60) AND fail-open, so if the secret is ever unset it lets ANY anonymous caller insert arbitrary `notifications` rows (any recipient_id, attacker-controlled title/body/data) via the service role ‚Äî i.e. mass phishing/spoofed in-app notifications and unbounded row insertion. push-dispatch (verify_jwt defaults true) can similarly be abused to spam FCM pushes to any user if deployed --no-verify-jwt.
  - _Fix:_ Make the gate fail-closed for functions that must be authenticated: if EDGE_SHARED_SECRET is unset, return 500/503 (`service not configured`) instead of allowing the request; and use a constant-time comparison. For send-notification specifically, require the secret unconditionally and consider keeping verify_jwt on with an internal caller using the service key.
- **Base profiles table grants SELECT on ALL columns (phone, email, bio, coords) to every authenticated user** `[security]` - `supabase/migrations/0001_init_auth_profiles.sql:64-65`
  - `create policy "profiles selectable by authenticated" on public.profiles for select to authenticated using (true)` makes every column of every profile readable by any logged-in user via the REST/PostgREST API ‚Äî including `phone`, `email`, `lat`, `lng`, and `bio`. The `profiles_public` view (L72-75) was clearly created as the 'safe projection' for recruiter/chat display, but it does NOT restrict the base table: a client can simply call `from('profiles').select('phone, email, lat, lng')` and bypass the projection entirely. The app's own first-party code only ever selects its own row from `profiles`, but RLS ‚Äî not client query shape ‚Äî is the security boundary, and here it is wide open. This is the root cause of the lat/lng leak above and additionally exposes phone/email PII platform-wide.
  - _Fix:_ Replace the blanket policy with a column-safe boundary. Practical options: (1) revoke direct SELECT on `public.profiles` from `authenticated` and route all cross-user reads through `profiles_public` (which itself must drop sensitive columns); or (2) split sensitive columns (phone, email, lat, lng) into a separate `profiles_private` table with `using (auth.uid() = id)` and keep only display-safe fields on `profiles`. Postgres RLS cannot do column-level filtering, so a view/separate-table split is required.
- **contact_info (home address, telegram, socials) is readable by every authenticated user** `[security]` - `supabase/migrations/0001_init_auth_profiles.sql:93-94`
  - `create policy "contact_info selectable by authenticated" on public.contact_info for select to authenticated using (true)` exposes each user's `address`, `telegram`, `website`, `linkedin`, and `github` to ANY logged-in user via `from('contact_info').select()`. The client only ever reads the signed-in user's own contact info (cv_repository.dart L268), so no first-party screen leaks it ‚Äî but an attacker with any account can enumerate every user's physical address and Telegram handle directly through the API. Combined with the lat/lng leak this compounds the physical-safety exposure.
  - _Fix:_ Scope the SELECT policy to the owner (`using (auth.uid() = profile_id)`), or to a relationship (e.g. only an employer with an application from this user, via an `is_job_owner`-style EXISTS check). If recruiter contact display is required, surface only non-sensitive fields through a dedicated view, never `address`.
- **Applicant can self-promote their own application to hired/offer via permissive status-history INSERT policy** `[security]` - `supabase/migrations/0003_applications_bookmarks.sql:76-82`
  - The `status history insert by applicant or owner` policy allows an INSERT when the row's application belongs to the caller as `a.applicant_id` OR when the caller is the job owner. Because `sync_application_status` (lines 114-124) denormalizes the latest inserted status onto `applications.current_status`, a job seeker can POST directly to `application_status_history` for their OWN application with `status='hired'` (or 'offer'/'shortlisted'/'interview') and flip their own pipeline state ‚Äî there is no constraint that pipeline-advancement statuses may only be set by the job owner. In a hiring marketplace this is a data-integrity / privilege-escalation hole: candidates can fabricate 'hired'/'offer' states that employers see. The focus contract states status updates must go through `application_status_history`, but the table is not access-segmented by role.
  - _Fix:_ Split the INSERT policy by who may set what: owner (`is_job_owner(application_id's job)`) may insert any status; applicant may insert ONLY a self-service status such as a future 'withdrawn' (or none). Encode the allowed status set per actor in the `with check` (e.g. `(public.is_job_owner(a.job_id)) OR (a.applicant_id = auth.uid() AND status IN ('withdrawn'))`). Also enforce `changed_by = auth.uid()` in the check so authorship can't be spoofed.
- **Applicant can directly UPDATE applications.current_status (second escalation path, no column guard)** `[security]` - `supabase/migrations/0003_applications_bookmarks.sql:65-67`
  - The `applications update by applicant or owner` policy is `for update ... using (auth.uid() = applicant_id or public.is_job_owner(job_id))` with NO WITH CHECK and NO column-level guard trigger. Postgres applies USING as the row check, so an applicant (auth.uid() = applicant_id) may UPDATE any column on their own application row, including `current_status`, straight to 'hired'/'offer' ‚Äî independent of the status_history path above. There is no trigger anywhere (grep confirms) preventing applicant writes to current_status. The applicant should only be allowed to edit cover_letter/answers (and ideally withdraw), never the pipeline status.
  - _Fix:_ Split the policy: give the applicant a constrained UPDATE (e.g. via a guard trigger that rejects any change to current_status when `not is_job_owner(NEW.job_id)`, mirroring guard_job_boost in 0011), and give the employer (is_job_owner) full UPDATE. Simplest: add a `guard_application_status` BEFORE UPDATE trigger that resets `NEW.current_status := OLD.current_status` unless `is_job_owner(NEW.job_id)`, so status only ever changes through the employer-driven history flow.
- **conversation_participants INSERT lets any user add themselves to an arbitrary conversation and read its messages** `[security]` - `supabase/migrations/0010_security_hardening.sql:28-30`
  - The hardening policy `participants insert self` is `for insert with check (profile_id = auth.uid())` ‚Äî it constrains only WHO is added (self), not WHICH conversation. So any authenticated user can `insert into conversation_participants (conversation_id, profile_id) values (<any conversation_id>, auth.uid())`, which makes `is_conversation_participant` return true and grants full read of that conversation's message history (and the realtime stream).
